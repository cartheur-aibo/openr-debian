# STTLOG Row 0400 Note: July 1, 2026

This note records the current strongest `STTLOG` row-level result in the
baseline MIND 2 simulator work.

## Why Row 0400 Matters

Under the current fatigue-first shutdown probe, row `0400` is the strongest
single current lever for whether the simulator defers shutdown once or accepts
shutdown immediately.

## Current Values

Observed current control values:

- specimen-style preserved tree:
  - `0400 0 0 0000 0110 00 01`
- bundled retail MIND 2 tree:
  - `0400 0 0 0000 0000 00 00`
- bundled retail MIND 3 tree:
  - `0400 0 0 0000 0000 00 00`
- packaged MIND 2 stick trees in this repo:
  - `0400 0 0 0000 0000 00 00`

So the current specimen-side `0400` state is not merely the normal baseline
MIND 2 value repeated elsewhere in the repo.

## Structural Clue

The token `0110` currently appears only in row `0400` among the checked
`STTLOG` files in this repo.

That makes the specimen-side row look less like a generic reused state pattern
and more like a specific latched condition or mode slot.

## Backward Reversion Result

When the specimen-style tree keeps all other current specimen-side state but
reverts only row `0400` back to the baseline value:

- `shutdown_resistance: 4 -> 2`
- the fatigue-first shutdown verdict changes from:
  - `defer power-off once`
  to:
  - `power-down accepted`

By contrast:

- reverting only `0300` does not flip the fatigue-first verdict
- reverting only `1000` does not flip the fatigue-first verdict
- reverting `0300 + 1000` together still does not flip the fatigue-first
  verdict

## Forward Transfer Result

The stronger causality check then moved specimen-side rows into a baseline tree.

Observed result:

- baseline control:
  - `shutdown_resistance=2`
  - verdict: accept shutdown
- baseline + specimen `0400` only:
  - `shutdown_resistance=4`
  - verdict: defer once
- baseline + specimen `0300` only:
  - `shutdown_resistance=2`
  - verdict: accept shutdown
- baseline + specimen `0300 + 0400 + 1000`:
  - `shutdown_resistance=4`
  - verdict: defer once

So within the present simulator heuristic:

- specimen-style `0400` is sufficient, by itself, to push a baseline tree over
  the fatigue-first shutdown boundary
- `0300` is not sufficient by itself

## Lean-Baseline Limit

That forward-transfer result does **not** mean `0400` is globally sufficient in
every baseline context.

When the same specimen-side `0400` was transplanted into the leaner packaged
MIND 2 stick tree under `features/aibo-mind2/build/stick`, the result was:

- baseline packaged tree:
  - `shutdown_resistance=1`
  - `social_attachment=5`
  - verdict: accept shutdown
- packaged baseline + specimen `0400` only:
  - `shutdown_resistance=3`
  - `social_attachment=5`
  - verdict: still accept shutdown
- packaged baseline + specimen `0400 + 1200`:
  - `shutdown_resistance=3`
  - `social_attachment=8`
  - verdict: defer once

So the stronger current reading is:

- `0400` is the primary shutdown-side lever
- but on leaner baselines it may still need `1200` to provide enough bridging
  social weight for the final shutdown deferral to happen
- `0046` can increase social weight further, but `1200` is the more important
  current companion row

## Current Interpretation Boundary

This is **not** proof of Sony's actual internal semantics for row `0400`.

It is a much narrower and more useful statement:

- row `0400` is the strongest currently isolated host-side shutdown pivot in
  the simulator
- the value difference is specimen-specific in the current repo dataset
- the fatigue-first probe is the scenario that exposes this pivot at the final
  verdict level
- row `0400` does not act alone in every baseline context; the current best
  companion row for crossing the verdict boundary on leaner baselines is
  `1200`
